# Updated Model Descriptions

## Model 1: Tiered Bundles

Fixed monthly pricing with predictable costs across five tiers (Free, Starter, Growth, Scale, Enterprise). Each tier includes set limits for profiles, events, emails, and SMS, with optional add-ons for advanced features. Best for businesses with consistent, predictable workloads who want easy budgeting and no surprises.

---

## Model 2: Pure Consumption (with Base Requirements)

Pay based on actual usage with no limits or caps. Features are grouped due to API dependencies (for now), but you're charged only for what you consume within those groups - no waste, no overages. Includes a free tier (500 profiles, 50K events, $50 monthly credit) with unlimited scalability beyond that. Best for variable or experimental workloads where you want maximum cost efficiency and flexibility without upfront commitments.

**Key Difference from Tiered:** No usage limits - pay per unit consumed, not fixed tier pricing.

---

## Model 3: Credits System (with Base Requirements)

Prepaid credit packages with volume discounts up to 40% that can be used flexibly across features. Features are grouped due to API dependencies (for now), but credits can be allocated however you need within those groups. Monthly free credits (50K) included, with four package tiers offering better rates at higher volumes. Best for high-volume users who want predictable costs with the flexibility to allocate usage across different features as needed.

**Key Difference from Tiered:** No usage limits - consume as many credits as you need, no caps.

---

## Quick Comparison

| Model | Pricing Type | Limits | Groupings | Best For |
|-------|--------------|--------|-----------|----------|
| **Tiered** | Fixed monthly | Hard caps per tier | Bundled by tier | Predictable workloads |
| **Consumption** | Pay-per-use | No limits | Required groups (for now) | Variable usage |
| **Credits** | Prepaid packages | No limits | Required groups (for now) | High-volume with budget control |

---

**Note:** "For now" indicates that feature groupings due to API dependencies may be decoupled in future iterations, moving toward truly à la carte pricing.
